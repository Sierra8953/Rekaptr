## Database Migration System

**Files:** `src/config.rs`, `src/db.rs`

**Problem:** Both `luma.db` and `history.db` use `CREATE TABLE IF NOT EXISTS` for schema setup. This works for the initial schema but breaks the moment a new column, table, or index needs to be added. Existing users' databases will silently have the wrong schema, causing runtime SQL errors or missing data.

**Implementation:**

### Migration runner
Add a generic migration function using SQLite's built-in `PRAGMA user_version`:
```rust
fn run_migrations(conn: &Connection, migrations: &[&str]) -> rusqlite::Result<()> {
    let current: u32 = conn.pragma_query_value(None, "user_version", |row| row.get(0))?;
    for (i, sql) in migrations.iter().enumerate() {
        let version = (i + 1) as u32;
        if version > current {
            conn.execute_batch(sql)?;
            conn.pragma_update(None, "user_version", version)?;
            log::info!("[DB] Applied migration v{}", version);
        }
    }
    Ok(())
}
```

### luma.db migrations (`src/config.rs`)
```rust
const LUMA_MIGRATIONS: &[&str] = &[
    // V1: Initial schema (IF NOT EXISTS for safe transition from v0.1.0)
    "CREATE TABLE IF NOT EXISTS config (id INTEGER PRIMARY KEY, json_data TEXT);
     CREATE TABLE IF NOT EXISTS favorites (clip_path TEXT PRIMARY KEY);",
];
```

Replace `init_db()` body with `run_migrations(&conn, LUMA_MIGRATIONS)`.

### history.db migrations (`src/db.rs`)
```rust
const HISTORY_MIGRATIONS: &[&str] = &[
    // V1: Initial schema
    "CREATE TABLE IF NOT EXISTS sessions (
        id INTEGER PRIMARY KEY,
        duration REAL NOT NULL,
        created_at INTEGER NOT NULL
    );",
];
```

Replace the `CREATE TABLE` in `GameDatabase::open()` with `run_migrations(&conn, HISTORY_MIGRATIONS)`.

### Key design decisions
- V1 uses `IF NOT EXISTS` so existing databases (user_version=0) upgrade safely
- No external crate needed — `PRAGMA user_version` is part of SQLite
- Each migration runs in a single `execute_batch` call (implicit transaction)
- Migration failures are propagated as errors, not silently swallowed

**Verification:**
- Fresh install: creates DB, runs V1, sets user_version=1
- Existing v0.1.0 DB: detects version=0, runs V1, tables already exist (IF NOT EXISTS), sets version=1
- Run app twice: second launch sees version=1, skips all migrations
- Add a test V2 migration: verify it runs only on version=1 databases

## Deterministic Segment File Operations

**File:** `src/ui/recording.rs`

### Remove post-shutdown sleep hacks

**Lines 17-18 (`toggle_recording_internal`):**
```rust
// BEFORE:
std::thread::spawn(move || {
    std::thread::sleep(std::time::Duration::from_millis(200));
    crate::utils::fixup_eos_segments(&game_dir);
});

// AFTER:
std::thread::spawn(move || {
    crate::utils::fixup_eos_segments(&game_dir);
});
```

**Line 58 (`stop_recording`):**
Same change — remove the `thread::sleep(200ms)` between `set_state(Null)` and `fixup_eos_segments`.

**Rationale:** `gst::State::Null` is a synchronous state transition in GStreamer. When `set_state(Null)` returns `Success`, all elements have released their resources including file handles. The 200ms sleep was added as insurance but is unnecessary and adds latency to the stop-recording flow.

### Reduce rename retry loop

**Lines 413-423 (`handle_fragment_closed`):**
```rust
// BEFORE: 25 retries x 200ms = 5 second max wait
for _ in 0..25 {
    cx.background_executor().timer(Duration::from_millis(200)).await;
    if std::fs::rename(&old_path, &new_path).is_ok() { return; }
}

// AFTER: Immediate attempt + 5 retries x 100ms = 500ms max wait
if std::fs::rename(&old_path, &new_path).is_ok() { return; }
for attempt in 1..=5 {
    cx.background_executor().timer(Duration::from_millis(100)).await;
    match std::fs::rename(&old_path, &new_path) {
        Ok(_) => return,
        Err(e) if attempt == 5 => {
            log::warn!("[Recording] Failed to rename segment after {} retries: {:?} — {}", attempt, old_path, e);
        }
        _ => {}
    }
}
```

**Rationale:** The `splitmuxsink-fragment-closed` signal fires after the fragment is finalized and the sink has moved on to the next fragment. The file should be immediately renameable. The retry exists only for Windows-specific edge cases (antivirus scanning, cloud sync like OneDrive). 5 retries at 100ms is more than sufficient.

**Verification:**
- Record 2 minutes, stop normally — all segments should have `_XXXXms.m4s` suffix
- Record and force-stop — segments should still be properly renamed by `fixup_eos_segments`
- Enable Windows Defender real-time scanning on the recording directory — renames should still succeed within the retry window
- Check logs for zero rename warnings during normal operation

## Hotkey Live-Reload

**Files:** `src/hotkeys.rs`, `src/ui/settings/hotkeys.rs`

**Problem:** `start_hotkey_listener` registers global hotkeys via `RegisterHotKey` once at startup and enters a `GetMessageW` loop forever. When the user changes hotkey bindings in settings, the new key codes are saved to config but the listener thread never re-registers them. The old hotkeys remain active until app restart.

**Implementation:**
1. Add a `mpsc::Sender<HotkeyCommand>` channel to `AppState`
2. The hotkey listener thread listens for `HotkeyCommand::Reload` messages between `GetMessageW` calls (use `MsgWaitForMultipleObjects` instead of blocking `GetMessageW` to check both Win32 messages and the channel)
3. On `Reload`: call `UnregisterHotKey` for all current hotkeys, read new config, call `RegisterHotKey` with new key codes
4. Settings UI sends `Reload` through the channel after saving config
5. Add feedback: if `RegisterHotKey` fails (key already registered by another app), show a toast

**Alternative (simpler):** Post a custom `WM_USER` message to the hotkey thread's message queue via `PostThreadMessage`. The listener checks for this message type and reloads.

**Verification:**
- Change the recording hotkey from F9 to F10 in settings — F10 should work immediately, F9 should stop working
- Set a hotkey that conflicts with another app (e.g., PrintScreen) — should show "hotkey already in use" error
- Rapid hotkey changes shouldn't crash or leak registrations

## Real Export Progress

**File:** `src/ui/export.rs:94-107`

**Problem:** The export progress bar is a fake timer that increments independently of ffmpeg. It reaches 100% on a fixed schedule regardless of actual export progress, file size, or encoding speed.

**Implementation:**
1. Spawn ffmpeg with `stderr` piped (it writes progress to stderr)
2. Parse the `time=HH:MM:SS.ms` field from ffmpeg's progress output
3. Calculate percentage as `parsed_time / total_duration * 100`
4. Send progress updates through a channel to the UI
5. Show indeterminate state until the first progress line is parsed
6. On ffmpeg exit, verify the output file exists and has nonzero size before showing success

**ffmpeg progress format:**
```
frame=  120 fps= 60 q=23.0 size=    1024kB time=00:00:02.00 bitrate=4194.3kbits/s speed=1.5x
```

Parse with a simple regex or string search for `time=` followed by the timestamp.

**Edge cases:**
- ffmpeg outputs progress at irregular intervals — update UI whenever a new line is available
- Hardware encoding is much faster than real-time — progress jumps quickly, that's fine
- If ffmpeg fails mid-export, show the actual error message from stderr, not just "export failed"

**Verification:**
- Export a 60-second clip — progress should correlate with actual encoding time
- Export a 5-second clip — should complete quickly, not wait for the fake timer
- Kill ffmpeg mid-export — should show an error, not "100% complete"

## Clip IN/OUT Ordering Enforcement

**File:** `src/ui/mod.rs:912-935`

**Problem:** `set_clip_in` and `set_clip_out` store timestamps independently without checking that `clip_start < clip_end`. If the user sets OUT at 5.0s then IN at 10.0s, `perform_export` passes `-ss 10.0 -to 5.0` to ffmpeg, which produces an empty file.

**Fix:**
```rust
pub fn set_clip_in(&mut self, cx: &mut Context<Self>) {
    self.clip_start = Some(current_position);
    // If OUT is set and is before IN, swap them
    if let (Some(start), Some(end)) = (self.clip_start, self.clip_end) {
        if start > end {
            std::mem::swap(&mut self.clip_start, &mut self.clip_end);
        }
    }
    cx.notify();
}
```

Apply the same logic in `set_clip_out`. Also add a visual indicator on the timeline showing which marker is IN vs OUT.

**Verification:**
- Set OUT at 5s, then IN at 10s — markers should swap so IN=5s, OUT=10s
- Export the clip — should contain the correct 5-second segment
- Set IN and OUT at the same timestamp — should show a "markers too close" warning

## Low Disk Space Toast Cooldown

**File:** `src/main.rs:262-313`

**Problem:** The disk monitor loop runs every 10 seconds. While free space is below 5 GB, it shows a new toast every iteration, flooding the user with notifications.

**Fix:** Add a `last_low_disk_warning: Option<Instant>` to the monitoring state. Only show the toast if no warning has been shown in the last 5 minutes, or if the disk crossed a new threshold (e.g., dropped below 1 GB after previously warning at 5 GB).

```rust
let should_warn = last_warning.map_or(true, |t| t.elapsed() > Duration::from_secs(300));
if free_gb < 5.0 && should_warn {
    // show toast
    last_warning = Some(Instant::now());
}
```

**Verification:**
- Set max buffer to fill disk below 5 GB — should see exactly one warning
- Wait 5 minutes — should see one more warning
- Disk drops below 1 GB — should warn immediately regardless of cooldown

## Logo Cache Eviction

**File:** `src/state.rs:252-265`

**Problem:** `evict_caches()` evicts `artwork_cache` and `portrait_cache` when they exceed `CACHE_MAX_ENTRIES` but skips `logo_cache`. Over many games, the logo cache grows unbounded.

**Fix:** Add `logo_cache` to the eviction logic in `evict_caches()`:
```rust
if self.logo_cache.len() > CACHE_MAX_ENTRIES {
    let excess = self.logo_cache.len() - CACHE_MAX_ENTRIES;
    let keys: Vec<String> = self.logo_cache.iter().take(excess).map(|e| e.key().clone()).collect();
    for key in keys { self.logo_cache.remove(&key); }
}
```

**Verification:**
- Load 200+ different game titles — cache size should plateau at `CACHE_MAX_ENTRIES`

**File:** `src/engine.rs:247-416`

**Problem:** `generate_pipeline_string()` assembles a GStreamer pipeline string without checking if the required elements exist. If an element is missing (e.g., user doesn't have NVIDIA GStreamer plugins), the error surfaces as an opaque `gst::parse::launch` failure with a message like "no element 'nvd3d11h264enc'".

**Implementation:**

            .ok_or_else(|| anyhow!("Required GStreamer plugin '{}' is not installed", name))?;
        Ok(())

### Update recording.rs
Change `generate_pipeline_string(...)` call to `PipelineBuilder::new(...)?.build()`. The `?` propagates the validation error, which the caller handles by showing a descriptive toast.
