# Rekaptr v0.4.0 — Performance

**Theme:** Eliminate unnecessary work, reduce latency, and make the app feel instant. Every interaction should respond within a single frame. Recording should have zero impact on the desktop.

**Target:** 2-3 weeks after v0.3.0

---

## Cache AppConfig In Memory

**Files:** `src/config.rs`, `src/state.rs`, all 60+ call sites

**Problem:** `AppConfig::load()` opens a new SQLite connection, runs `init_db()`, queries the config row, and deserializes JSON — every single time. This happens ~60 times/second during video playback because `get_current_audio_tracks()` is called every render frame. Each call involves:
1. `Connection::open()` — file open + SQLite header read
2. `init_db()` — two `CREATE TABLE IF NOT EXISTS` statements
3. `SELECT json_data FROM config WHERE id = 1`
4. `serde_json::from_str::<AppConfig>()` — full JSON deserialization
5. `Connection::drop()` — fsync + file close

On HDDs or with antivirus, this causes visible frame drops during timeline scrubbing.

**Implementation:**

### Add cached config to AppState
```rust
// In src/state.rs
pub struct AppState {
    // ...
    pub config_cache: parking_lot::RwLock<AppConfig>,
}
```

### Modify AppConfig API
```rust
// New: read from cache (fast path)
impl AppConfig {
    pub fn cached(app_state: &AppState) -> AppConfig {
        app_state.config_cache.read().clone()
    }

    // Existing save() also updates cache
    pub fn save(&self, app_state: &AppState) {
        *app_state.config_cache.write() = self.clone();
        // ... existing SQLite write
    }
}
```

### Migration path
- Phase 1: Add the cache, update hot paths (`get_current_audio_tracks`, `get_storage_root`, render functions) to use the cache
- Phase 2: Audit remaining `AppConfig::load()` calls and migrate them
- Phase 3: Make `AppConfig::load()` private, force all reads through the cache

### Call sites to migrate (priority order)
1. `src/ui/mod.rs:800` — `get_current_audio_tracks()` — called every frame
2. `src/utils.rs:50` — `get_storage_root()` — called from cleanup thread
3. `src/ui/dashboard.rs` — multiple render-time config reads
4. `src/ui/settings/*.rs` — settings panels (lower priority, not hot)

**Verification:**
- Profile with `tracy` or manual timing: measure time spent in `AppConfig::load()` before and after
- Timeline scrubbing should be noticeably smoother on systems with slow storage
- Change a setting, verify it takes effect immediately (cache is updated on save)
- Background cleanup thread should not open SQLite connections during operation

## Eliminate Double scan_segments

**File:** `src/utils.rs`

**Problem:** `generate_session_playlist()` (line ~596) calls `scan_segments()` to get the segment list, then passes it to `generate_master_playlist()` (line ~607) which calls `scan_segments()` again internally. Each call does:
1. `std::fs::read_dir()` on the game directory
2. For each `.m4s` file: read filename, parse index, call `metadata()` for modification time
3. Build and sort a `BTreeMap<u64, PathBuf>`

For a game with 1000 segments (common after a few hours of recording), this is 2000 unnecessary syscalls.

**Fix:** Change `generate_master_playlist` to accept the segment map as a parameter instead of scanning internally:

```rust
// BEFORE:
pub fn generate_master_playlist(game_dir: &Path) -> String {
    let segments = scan_segments(game_dir);
    // ...
}

// AFTER:
pub fn generate_master_playlist(game_dir: &Path, segments: &BTreeMap<u64, PathBuf>) -> String {
    // ... use segments directly
}
```

Update `generate_session_playlist` to pass the scan results through.

**Verification:**
- Record a long session (500+ segments), load the timeline — should be measurably faster
- Playlist content should be identical before and after the change

## Targeted Process Refresh

**Files:** `src/engine.rs:193-228, 418-483`

**Problem:** Both `resolve_pid()` and `start_app_capture()` create `System::new_all()` from the `sysinfo` crate, which:
1. Enumerates every running process on the system
2. Reads CPU usage, memory, disk I/O, and open files for each
3. Builds a full process tree

On a system with 300+ processes, this takes 50-200ms. It's called at recording start time, adding noticeable delay.

**Fix:** Use targeted refresh that only reads process names and PIDs:

```rust
let mut sys = System::new();
sys.refresh_processes_specifics(
    sysinfo::ProcessesToUpdate::All,
    true, // update existing only
    sysinfo::ProcessRefreshKind::new(), // minimal: just name and PID
);
```

For `start_app_capture` which needs parent PID info, add only the needed fields:
```rust
sysinfo::ProcessRefreshKind::new().with_parent()
```

**Verification:**
- Measure recording start latency before and after
- Process matching still works correctly for all source types (game, monitor, app)

## Window-Based Process Targeting

**File:** `src/engine.rs:193-228`

**Problem:** `resolve_pid()` falls back to picking the process with the highest memory usage among name matches. This breaks for:
- Multi-process games (Chromium-based games have many processes)
- Games with separate launcher processes that use more memory than the game
- Games where the rendering process is a child with less memory than the parent

**Implementation:**
After collecting matching PIDs by process name, prefer the one that owns a visible window:

```rust
fn find_pid_with_visible_window(pids: &HashSet<u32>) -> Option<u32> {
    use windows::Win32::UI::WindowsAndMessaging::*;
    use windows::Win32::Foundation::*;
    use std::sync::Mutex;

    struct EnumState {
        target_pids: HashSet<u32>,
        found_pid: Option<u32>,
    }

    let state = Arc::new(Mutex::new(EnumState {
        target_pids: pids.clone(),
        found_pid: None,
    }));
    let state_ptr = Arc::into_raw(state.clone());

    unsafe extern "system" fn callback(hwnd: HWND, lparam: LPARAM) -> BOOL {
        let state = &*(lparam.0 as *const Mutex<EnumState>);
        if IsWindowVisible(hwnd).as_bool() {
            let mut pid = 0u32;
            GetWindowThreadProcessId(hwnd, Some(&mut pid));
            let mut s = state.lock().unwrap();
            if s.target_pids.contains(&pid) {
                s.found_pid = Some(pid);
                return FALSE; // stop enumeration
            }
        }
        TRUE // continue
    }

    unsafe {
        let _ = EnumWindows(Some(callback), LPARAM(state_ptr as isize));
        Arc::from_raw(state_ptr); // reclaim Arc
    }

    let result = state.lock().unwrap().found_pid;
    result
}
```

Integrate into `resolve_pid()`: try window-based detection first, fall back to highest-memory.

**Verification:**
- Launch a Chromium-based game (e.g., from Epic/Steam overlay) — should pick the renderer process, not the launcher
- Launch a game from a launcher that stays open — should pick the game, not the launcher
- Test with `resolve_pid("explorer")` — should return explorer.exe's PID (always has visible windows)

## Async Sessions View

**File:** `src/ui/sessions.rs:8`

**Problem:** `render_sessions()` calls `fetch_all_sessions()` synchronously on every render, which does:
1. `read_dir` on the storage root
2. For each game directory: open `history.db`, query all sessions
3. For each session: scan segment files, potentially call ffprobe for duration
4. Sort and return

This blocks the UI thread. With many games and sessions, the sessions tab can freeze for seconds.

**Fix:** Follow the same pattern as `refresh_clips()`:
1. Add `cached_sessions: Vec<Session>` and `is_loading_sessions: bool` to `RekaptrWorkspace`
2. Add `refresh_sessions()` that spawns an async task to call `fetch_all_sessions()`
3. `render_sessions()` reads from the cache, shows a loading spinner while refreshing
4. Trigger refresh when navigating to the sessions tab and after recording stops

**Verification:**
- Navigate to sessions tab — should show cached data immediately, not freeze
- Add 20+ game directories with sessions — tab should still be responsive
- Start/stop recording — sessions list should update automatically

## Track and Cancel Background Tasks

**Files:** Multiple (18+ `.detach()` calls across the codebase)

**Problem:** Every `cx.spawn(...).detach()` and `std::thread::spawn(...)` creates a fire-and-forget task. Tasks are never tracked, cancelled, or joined. Specific issues:
- Audio capture threads (`src/ui/recording.rs:257`) loop forever until `push_buffer` fails
- Segment rename tasks (`src/ui/recording.rs:413`) can pile up (one per segment)
- Artwork download tasks can pile up if the user scrolls through many games quickly

**Implementation:**

### Phase 1: Audio capture thread tracking
```rust
// In AppState or recording state:
pub audio_capture_handles: Mutex<Vec<std::thread::JoinHandle<()>>>,

// In setup_audio_feeders: store handles
let handle = std::thread::spawn(move || { ... });
self.app_state.audio_capture_handles.lock().push(handle);

// In stop_recording: signal and join
// Set a shared AtomicBool to signal threads to exit
self.app_state.audio_capture_stop.store(true, Ordering::SeqCst);
for handle in self.app_state.audio_capture_handles.lock().drain(..) {
    let _ = handle.join();
}
```

### Phase 2: Artwork download deduplication
Use a `HashSet<String>` of in-flight game titles. Skip spawning a new download if one is already in progress for that title.

### Phase 3: Task abort on workspace drop
Store `Task` handles from `cx.spawn()` in a `Vec<Task<()>>`. On workspace teardown, drop the vec to cancel pending tasks.

**Verification:**
- Start/stop recording 20 times rapidly — thread count should not grow
- Scroll through 100 games in clips view — artwork download threads should be bounded
- Check thread count in Task Manager during normal operation — should be stable
