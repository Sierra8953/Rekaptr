## Move Config to %APPDATA%

**Files:** `src/config.rs:314-318`, installer configuration

**Problem:** `get_db_path()` stores `luma.db` next to the executable. If installed to `C:\Program Files\`, writes require admin privileges and silently fail (`let _ = conn.execute(...)` discards errors). Even in user-writable locations, storing config next to the exe is non-standard on Windows and breaks with auto-updaters that replace the exe directory.

**Implementation:**
1. Primary path: `%APPDATA%\Rekaptr\luma.db` (e.g., `C:\Users\user\AppData\Roaming\Rekaptr\luma.db`)
2. Create the directory on first run: `std::fs::create_dir_all()`
3. Migration: on startup, if the new path doesn't exist but the old exe-relative path does, copy the old DB to the new location
4. Update `get_db_path()`:
```rust
pub fn get_db_path() -> PathBuf {
    if let Some(appdata) = std::env::var_os("APPDATA") {
        let dir = PathBuf::from(appdata).join("Rekaptr");
        let _ = std::fs::create_dir_all(&dir);
        return dir.join("luma.db");
    }
    // Fallback to exe-relative (portable mode)
    std::env::current_exe()
        .ok()
        .and_then(|p| p.parent().map(|d| d.join("luma.db")))
        .unwrap_or_else(|| PathBuf::from("luma.db"))
}
```
5. Consider a `portable.txt` marker file next to the exe that forces exe-relative storage (for USB/portable installs)

**Verification:**
- Fresh install: config created in `%APPDATA%\Rekaptr\`
- Upgrade from v0.4.0: old config copied to new location, app works with existing settings
- Install to `C:\Program Files\Rekaptr\`: config still works (goes to APPDATA, not Program Files)
- Portable mode: place `portable.txt` next to exe, config stays exe-relative

## Comprehensive Error Reporting

**Files:** Multiple — audit all `let _ =` patterns

**Problem:** Many operations silently discard errors:
- `config.save()` — SQLite write failures are ignored (`src/config.rs:392`)
- `pipeline.set_state()` — state change failures discarded in multiple places
- `std::process::Command::new("explorer")` — file explorer launch failures ignored
- `conn.execute()` — database operations fail silently throughout

Users experience mysterious behavior (settings not saving, clips not opening) with no indication of what went wrong.

**Implementation:**

### Tier 1: Critical operations — show toast on failure
- Config save: "Settings could not be saved — disk may be full"
- Pipeline state changes: "Recording failed to start — [specific error]"
- Export ffmpeg failures: "Export failed — [ffmpeg error message]"
- Database operations: "Database error — [details]"

### Tier 2: Non-critical operations — log warning
- File explorer launch failures
- Artwork download failures
- Thumbnail generation failures
- Tray icon updates

### Tier 3: Expected failures — silent (keep as-is)
- Network requests for Steam artwork (offline is normal)
- Hotkey registration conflicts (already handled separately in v0.3.0)

### Pattern
Replace `let _ = risky_operation()` with:
```rust
if let Err(e) = risky_operation() {
    log::warn!("[Context] Operation failed: {}", e);
    // For tier 1: also show toast
}
```

**Verification:**
- Make disk read-only, change a setting — should see "settings could not be saved" toast
- Corrupt the database file — should see a clear error on startup
- Remove ffmpeg.exe, try to export — should see "ffmpeg not found" error

## Game Title Collision Handling

**File:** `src/utils.rs:43-48`

**Problem:** `clean_title()` strips all non-alphanumeric characters and lowercases, so:
- "DOOM" and "D.O.O.M" → `"doom"` (same directory)
- "Halo 3" and "Halo3" → `"halo3"` (same directory)
- "GTA V" and "GTA5" → different (`"gtav"` vs `"gta5"`)

Games with colliding titles share a recording directory, mixing their footage.

**Implementation:**
1. Keep the current sanitization (it's needed for filesystem safety)
2. Add a disambiguation suffix when a collision is detected:
```rust
pub fn get_game_directory(title: &str) -> PathBuf {
    let base = clean_title(title);
    let root = get_storage_root();
    let dir = root.join(&base);

    // Check if this directory is already claimed by a different title
    let marker = dir.join(".game_title");
    if dir.exists() {
        if let Ok(existing_title) = std::fs::read_to_string(&marker) {
            if existing_title.trim() != title {
                // Collision — append a hash suffix
                let hash = crc32fast::hash(title.as_bytes());
                return root.join(format!("{}_{:08x}", base, hash));
            }
        }
    }

    // Claim this directory
    let _ = std::fs::create_dir_all(&dir);
    let _ = std::fs::write(&marker, title);
    dir
}
```
3. The `.game_title` marker file records which exact title owns the directory

**Verification:**
- Two games with colliding clean titles get separate directories
- Existing recordings for "DOOM" stay in the "doom" directory
- The marker file survives across sessions

## Smart Clip Library Grouping

**File:** `src/ui/clips.rs`, `src/ui/mod.rs`

**Problem:** The clips library shows "MOST RECENT" (last 4 clips) then "GAMES" (grouped by title). There's no temporal context — you can't tell if a clip is from today or six months ago.

**Implementation:**
Add time-based section headers in `rebuild_library_items()`:
- "TODAY" — clips from the current calendar day
- "YESTERDAY"
- "THIS WEEK" — last 7 days excluding today/yesterday
- "THIS MONTH" — last 30 days excluding this week
- "OLDER"

Within each section, clips are still sorted newest-first. Favorites section stays at the top.

```rust
fn time_section(timestamp: u64) -> &'static str {
    let now = SystemTime::now().duration_since(UNIX_EPOCH).unwrap().as_secs();
    let age = now.saturating_sub(timestamp);
    let days = age / 86400;
    match days {
        0 => "TODAY",
        1 => "YESTERDAY",
        2..=6 => "THIS WEEK",
        7..=29 => "THIS MONTH",
        _ => "OLDER",
    }
}
```

Only show sections that have clips. The game-grouped view (when clicking into a specific game) keeps its current flat layout.

**Verification:**
- Record a clip today — appears under "TODAY"
- Older clips appear in the correct sections
- Empty sections are not shown
- Switching to game-filtered view still works
