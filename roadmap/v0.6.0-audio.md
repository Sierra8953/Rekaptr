## Audio Waveform on Timeline

**Files:** `src/ui/dashboard.rs` (timeline rendering), `src/engine.rs` (audio level data)

**Problem:** The timeline shows video position but gives no visual feedback about audio. Users can't tell if audio is being captured, where silence gaps are, or where loud moments (highlights) happened.

**Implementation:**

### Data collection during recording
The GStreamer pipeline already has a `level` element on the system audio track (`wasapi2src loopback=true ... ! level interval=100000000 post-messages=true`). This posts bus messages every 100ms with peak and RMS levels per channel.

1. In `spawn_bus_monitor`, handle `level` element messages:
```rust
gst::MessageView::Element(msg) => {
    if let Some(s) = msg.structure() {
        if s.name() == "level" {
            let peak = s.get::<gst::List>("peak").ok();
            let rms = s.get::<gst::List>("rms").ok();
            // Store in a ring buffer or Vec<f32> indexed by time
        }
    }
}
```

2. Store waveform data in `RecordingStats` or a dedicated struct:
```rust
pub struct AudioWaveform {
    /// RMS levels sampled at 10Hz (one per 100ms)
    pub samples: Vec<f32>,
    /// Duration covered per sample
    pub sample_interval: Duration,
}
```

### Waveform rendering on timeline
1. Below (or behind) the timeline progress bar, render the waveform as a series of vertical bars
2. Scale bar heights by RMS level (dBFS → linear → pixel height)
3. Color code: green for normal levels, yellow for hot (-6dBFS to -3dBFS), red for clipping (above -1dBFS)
4. Render only the visible portion of the waveform (based on current zoom level)

### For playback (clip preview)
When loading a clip for preview, generate the waveform offline:
1. Use ffprobe or ffmpeg to extract audio levels: `ffmpeg -i clip.mp4 -af astats=metadata=1:reset=4800 -f null -`
2. Parse the per-frame RMS values from stderr
3. Cache the waveform data alongside the clip

**Verification:**
- Record with system audio — waveform should show activity during loud moments
- Record silence — waveform should be flat
- Scrub the timeline — waveform should scroll with it
- Open a clip in preview — waveform should load and display

## Noise Gate

**Files:** `src/audio.rs`, `src/config.rs`

**Problem:** RNNoise denoising is available but a simple noise gate (silence suppression) is often more appropriate for gaming. A noise gate cuts audio below a threshold, which is less aggressive than AI denoising and introduces zero latency.

**Implementation:**

### Config
Add to `MicSettings`:
```rust
pub noise_gate_enabled: bool,
pub noise_gate_threshold_db: f32,  // e.g., -40.0 dB
pub noise_gate_attack_ms: f32,     // e.g., 1.0 ms
pub noise_gate_release_ms: f32,    // e.g., 100.0 ms
```

### DSP in MicProvider
In the `new_sample` callback (`src/audio.rs`), after the optional RNNoise pass:
```rust
if mic_settings.noise_gate_enabled {
    let threshold = 10f32.powf(mic_settings.noise_gate_threshold_db / 20.0);
    for sample in buffer.iter_mut() {
        let level = sample.abs();
        if level < threshold {
            // Apply release envelope (smooth fade to zero)
            gate_gain = (gate_gain - release_step).max(0.0);
        } else {
            // Apply attack envelope (smooth fade to full)
            gate_gain = (gate_gain + attack_step).min(1.0);
        }
        *sample *= gate_gain;
    }
}
```

### Settings UI
Add noise gate controls to the microphone settings panel:
- Enable/disable toggle
- Threshold slider (-60dB to -10dB)
- Attack time slider (0.1ms to 10ms)
- Release time slider (10ms to 500ms)
- Live visualization: show where the gate threshold sits relative to the current mic level

**Verification:**
- Enable noise gate at -30dB — keyboard noise should be suppressed, voice should pass through
- Adjust threshold while monitoring — should hear the gate open and close
- Attack and release should sound smooth, no clicking or abrupt cuts
- Compare recording with gate vs without — gate should produce cleaner audio

## Push-to-Talk Improvements

**Files:** `src/hotkeys.rs`, `src/audio.rs`, `src/ui/dashboard.rs`

**Problem:** Push-to-talk exists but has no visual feedback — users can't tell if it's active, and there's no indicator when the mic is hot.

**Implementation:**

### Visual indicator
1. When PTT key is held, show a mic icon overlay in the recording panel (pulsing red dot)
2. When PTT key is released, the indicator fades out over 200ms
3. In the system tray tooltip, show "Mic: Active" or "Mic: Push-to-Talk (F11 to speak)"

### Configurable mode
Add PTT mode options:
- **Hold-to-talk:** Mic active only while key is held (current behavior)
- **Toggle:** Press once to unmute, press again to mute
- **Voice-activated with PTT override:** Combine noise gate with manual override

### Latency reduction
Current PTT implementation may have latency from the hotkey polling interval (50ms in `src/main.rs:352`). For PTT:
1. Reduce the PTT-specific polling to 10ms
2. Or use a raw input hook (`WM_INPUT`) for the PTT key specifically, which is lower latency than `RegisterHotKey`

**Verification:**
- Hold PTT key — mic indicator appears immediately, voice is captured
- Release PTT key — indicator fades, mic goes silent
- Toggle mode — press once to start, press again to stop
- Measure PTT latency — should be under 20ms from keypress to audio capture

## Mic Monitoring (Listen to Yourself)

**Files:** `src/audio.rs`, `src/ui/settings/audio.rs`

**Problem:** The settings panel has a mic test button but no way to monitor your mic in real-time while recording. Users can't hear themselves to verify audio quality, noise suppression effectiveness, or PTT behavior.

**Implementation:**

### Audio loopback
1. Create a WASAPI render client targeting the default output device
2. In the `MicProvider` push loop, after processing (denoise, gate), also write the processed audio to the render client
3. Add a configurable monitoring volume (default: muted to prevent feedback)
4. Add a toggle in the recording panel: "Monitor Mic" with a headphone icon

### Feedback prevention
- Default monitoring to OFF
- Show a warning when enabling: "Use headphones to prevent feedback"
- If the selected mic device matches the output device (e.g., same USB headset), reduce monitoring volume to prevent feedback loops
- Add a simple feedback detection: if the mic input suddenly matches the monitoring output, auto-mute monitoring

**Verification:**
- Enable monitoring with headphones — should hear your processed mic audio in real-time
- Adjust mic volume, noise gate, denoise — should hear changes immediately
- Enable monitoring with speakers — should show feedback warning
- PTT + monitoring — should only hear yourself while PTT is held

## Audio Mixer Panel

**Files:** `src/ui/dashboard.rs` (or new `src/ui/mixer.rs`)

**Problem:** Audio track configuration is buried in settings. During recording, users can't adjust individual track volumes without opening settings and losing focus on the recording.

**Implementation:**

### Floating mixer panel
1. Add a "Mixer" button to the recording panel that opens a floating overlay
2. Each audio track gets a vertical fader with:
   - Track name and type icon (speaker/mic/app)
   - Volume slider (0-150%)
   - Mute button
   - Live level meter (from the levels work above)
   - Solo button (mute all others)
3. Changes apply in real-time by updating the `volume` element in the GStreamer pipeline:
```rust
if let Some(vol) = pipeline.by_name(&format!("vol_{}", track_index)) {
    vol.set_property("volume", new_volume);
}
```
