## Instant Replay / Replay Buffer

**Files:** `src/engine.rs`, `src/ui/recording.rs`, `src/ui/mod.rs`, `src/hotkeys.rs`

**Problem:** Currently, Rekaptr records continuously and saves clips manually. Many users want a "ShadowPlay-style" replay buffer — always recording the last N minutes, with a hotkey to save the recent buffer as a clip without having continuous recording management.

**Implementation:**

### How it works
1. Rekaptr already records in 6-second segments via `splitmuxsink`. The replay buffer is the last N minutes of segments.
2. The buffer management already exists (`start_buffer_cleanup_thread` in `src/utils.rs`) — it deletes segments older than the configured buffer size.
3. "Save replay" = copy the last N segments to a "saved clips" directory and concatenate into a single MP4.

### New config
```rust
pub struct ReplaySettings {
    pub enabled: bool,
    pub duration_seconds: u32,  // 30, 60, 120, 300, 600
    pub auto_start: bool,       // Start replay buffer on app launch
}
```

### Hotkey: Save Replay
1. Register a new global hotkey (default: F10, configurable)
2. On trigger:
   - Calculate how many segments cover the requested duration
   - Copy those segments to a timestamped directory under `Clips/`
   - Spawn ffmpeg to concatenate the segments into a single `.mp4`
   - Show a toast: "Replay saved — last 2 minutes"
3. The recording continues uninterrupted — saving a replay doesn't stop the buffer

### UI
- Add a "Replay Buffer" toggle to the dashboard
- Show buffer status: "Replay: 2:00 buffered" with a circular progress ring
- Hotkey shown as a hint: "Press F10 to save replay"
- After saving, show the clip in the notification: "Replay saved" with a "View" button

### Technical details
- The replay buffer IS the normal recording — just with automatic cleanup enabled
- Saving a replay copies segments (not moves) so the buffer continues
- Concatenation uses ffmpeg's concat demuxer for speed (no re-encoding):
  ```
  ffmpeg -f concat -safe 0 -i segments.txt -c copy replay_2024_01_15_14_30.mp4
  ```
- Segment list file is generated dynamically from the last N segments

**Verification:**
- Enable replay buffer, play a game for 5 minutes, press save hotkey — should get a clip of the last N minutes
- Save replay multiple times — each should be independent
- Buffer should not grow beyond configured duration
- Saving a replay should not cause any frame drops in the recording

## Recording Pause/Resume

**Files:** `src/ui/recording.rs`, `src/state.rs`, `src/hotkeys.rs`

**Problem:** Users currently have to stop and start a new recording session if they want to skip a portion (loading screen, menu, AFK break). This creates separate clips that need manual concatenation.

**Implementation:**

### GStreamer approach
GStreamer supports the `Paused` state, but pausing `d3d11screencapturesrc` can cause buffer starvation and timestamp discontinuities on resume. Safer approach:

1. **On pause:** Drop all incoming buffers using an `identity` element with `drop=true`, or disconnect the screen capture source from the pipeline
2. **On resume:** Reconnect and let `splitmuxsink` start a new fragment with the correct timestamp offset

Alternative (simpler): Use `splitmuxsink`'s fragment signaling to track pause timestamps, and mark paused sections as discontinuities in the HLS playlist. The paused segments exist but are empty or very small.

### State machine update
```rust
pub enum RecordingPhase {
    Idle,
    Starting,
    Recording,
    Paused,      // NEW
    Stopping,
}
```

### Hotkey
- Default: `Ctrl+F9` (pause/resume)
- Or double-tap the recording hotkey

### UI
- Paused state shows a blinking "PAUSED" indicator where the timer normally is
- Timer freezes during pause, resumes on unpause
- Timeline shows paused regions as grayed-out gaps

### Segment handling
When paused:
1. Send a `force-key-unit` event to finalize the current segment
2. Stop feeding video frames to the pipeline (mute the source)
3. Continue running audio capture (optional — some users want to keep mic audio during pause)

When resumed:
1. Resume feeding video frames
2. The `splitmuxsink` starts a new segment automatically
3. Add a `#EXT-X-DISCONTINUITY` marker in the playlist at the pause point

**Verification:**
- Record, pause for 10 seconds, resume, stop — playback should skip the paused section seamlessly
- Pause during a game, alt-tab, resume — no corruption or timestamp issues
- Paused segments should be minimal size (no video data)
- Total recording duration should exclude paused time

## Multi-Monitor Support

**Files:** `src/engine.rs`, `src/ui/add_source.rs`, `src/game_detector.rs`

**Problem:** `d3d11screencapturesrc` captures a single output. Users with multiple monitors can only record one at a time, and monitor selection is tied to the adapter index which may not be intuitive.

**Implementation:**

### Monitor enumeration
Use the Windows `EnumDisplayMonitors` API to get:
- Monitor name/model
- Resolution and position
- Primary/secondary designation
- DPI scaling

Display in the source selection dropdown:
```
[Monitor 1 — Dell U2720Q (3840x2160) — Primary]
[Monitor 2 — ASUS VG248 (1920x1080)]
[All Monitors (5760x2160)]
```

### Single monitor capture
For individual monitors, map the monitor's rectangle to the correct `d3d11screencapturesrc` parameters:
- `adapter` index for the GPU driving that output
- For WGC (Windows Graphics Capture): use `monitor-handle` instead of `window-handle`
- Alternatively, use DXGI output duplication with the specific output index

### Multi-monitor capture (all monitors)
Capture the entire virtual desktop:
1. Use `d3d11screencapturesrc capture-api=dxgi` without a specific output — captures the primary output
2. For true multi-monitor: use Desktop Duplication API to capture each output, then composite them into a single frame using `compositor` or `d3d11compositor`
3. This requires careful handling of different resolutions and DPI scales

### Initial scope (v0.7.0)
- Single monitor selection from a friendly dropdown (not adapter index)
- Defer full multi-monitor compositing to v0.8.0 or later

**Verification:**
- Two monitors at different resolutions — each selectable independently
- Selected monitor changes without app restart
- Monitor disconnect during recording — graceful fallback or error, not crash
- Hot-plugging a monitor updates the dropdown

## Custom Overlays

**Files:** New `src/overlay.rs`, `src/engine.rs`

**Problem:** Users want to include webcam footage, a timer, or a watermark in their recordings without using OBS.

**Implementation:**

### Webcam overlay
1. Add a webcam source using GStreamer's `ksvideosrc` (Kernel Streaming) or `mfvideosrc` (Media Foundation)
2. Composite onto the screen capture using `d3d11compositor` or `compositor`:
```
d3d11screencapturesrc ! compositor name=comp
  mfvideosrc ! video/x-raw,width=320,height=240 ! comp.
```
3. User-configurable: position (corner), size, shape (rectangle/circle), border

### Timer overlay
1. Use GStreamer's `clockoverlay` or `textoverlay` element
2. Show elapsed recording time, or current time of day
3. Configurable position, font, color, background

### Image watermark
1. Use `gdkpixbufoverlay` or load a PNG and composite with `compositor`
2. Support transparency
3. Configurable position, size, opacity

### Configuration UI
Add an "Overlays" section to the recording settings:
- Toggle each overlay type on/off
- Preview the overlay composition before recording
- Drag-to-position on a preview of the capture area

### Initial scope (v0.7.0)
- Timer overlay (simplest, highest demand)
- Static image watermark
- Defer webcam to v0.8.0 (requires significant pipeline changes)

**Verification:**
- Enable timer overlay — visible in the recording, correct position
- Enable watermark — visible with correct transparency
- Overlays should not affect recording performance (< 1% FPS impact)
- Overlays should be visible in playback and export

## Scheduled Recording

**Files:** New `src/scheduler.rs`, `src/ui/settings/scheduler.rs`

**Problem:** Users who stream on a schedule, or who want to capture specific game sessions automatically, need to set up recording in advance.

**Implementation:**

### Schedule types
1. **One-time:** Record from 3:00 PM to 5:00 PM on March 15
2. **Recurring:** Every Tuesday and Thursday from 7:00 PM to 10:00 PM
3. **Game-triggered:** Start recording when [Game.exe] launches, stop when it exits (already partially implemented via auto-record)

### Scheduler engine
```rust
pub struct ScheduleEntry {
    pub id: u64,
    pub source: String,
    pub schedule_type: ScheduleType,
    pub start_time: chrono::NaiveDateTime,
    pub end_time: Option<chrono::NaiveDateTime>,
    pub recurrence: Option<Recurrence>,
    pub enabled: bool,
}

pub enum ScheduleType {
    OneTime,
    Recurring,
    GameLaunch { exe_name: String },
}
```

### Background monitor
A dedicated async task checks every 30 seconds:
1. Is there a schedule entry whose start time has passed and end time hasn't?
2. If yes and not already recording, start recording for that source
3. If the end time passes, stop recording
4. For game-launch schedules, check if the target process is running

### UI
Add a "Schedule" tab in settings:
- Calendar-style view of upcoming recordings
- Add/edit/delete schedule entries
- Visual timeline showing when recordings will occur
- Status: "Next scheduled recording: Tuesday 7:00 PM — Fortnite"

### Initial scope (v0.7.0)
- Game-triggered auto-record (improve existing implementation)
- One-time scheduled recording
- Defer recurring schedules to later (requires date/time picker UI component)

**Verification:**
- Schedule a recording for 1 minute from now — should start automatically
- Set end time — should stop automatically
- Schedule overlapping with manual recording — should show a conflict warning
- App restart should preserve schedules (stored in luma.db)

## Recording Profiles

**Files:** `src/config.rs`, `src/ui/settings/*.rs`

**Problem:** Users want different settings for different scenarios: high quality for edited content, smaller files for Discord sharing, low-impact for competitive gaming. Currently they manually change settings each time.

**Implementation:**

### Profile system
```rust
pub struct RecordingProfile {
    pub name: String,
    pub video: VideoSettings,
    pub audio_tracks: Vec<AudioRouting>,
    pub mic: MicSettings,
    pub overlays: Vec<OverlayConfig>,
}
```

### Built-in profiles
- **Quality:** Native resolution, AV1, CQ 20, all audio tracks
- **Performance:** 1080p, H.264, CBR 15Mbps, system audio only
- **Compact:** 720p, H.264, CBR 5Mbps, system audio only (for Discord/sharing)
- **Custom:** User-defined

### UI
- Profile selector dropdown in the recording panel (next to the record button)
- Settings panels show which profile is being edited
- "Save as Profile" button in settings
- Per-game profile assignment in the game registry

**Verification:**
- Switch profiles — settings should change immediately
- Start recording with a profile — correct settings applied
- Delete a profile — falls back to default
- Per-game profile overrides global profile
