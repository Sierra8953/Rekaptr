## Smart Cut (Keyframe-Aware Trimming)

**Files:** `src/ui/export.rs`, `src/utils.rs`

**Problem:** The current export always re-encodes the trimmed clip. For a 30-second clip from a 1080p60 AV1 recording, re-encoding takes 10-60 seconds depending on hardware. Most users just want to trim the start and end — they don't need quality changes.

**Implementation:**

### How it works
1. Find the nearest keyframe before the IN point and after the OUT point
2. Copy all packets between those keyframes without re-encoding (`-c copy`)
3. Only re-encode the partial GOPs at the start and end
4. Concatenate: re-encoded start + copied middle + re-encoded end

This is the technique used by LosslessCut and is 10-100x faster than full re-encode for most clips.

### ffmpeg implementation
```bash
# Step 1: Find keyframes
ffprobe -select_streams v -show_frames -show_entries frame=pkt_pts_time,key_frame \
  -of csv input.mp4

# Step 2: Copy middle (between nearest keyframes)
ffmpeg -ss {keyframe_before_in} -to {keyframe_after_out} -i input.mp4 -c copy middle.mp4

# Step 3: Re-encode head (IN point to first keyframe)
ffmpeg -ss {in_point} -to {keyframe_before_in} -i input.mp4 -c:v libx264 head.mp4

# Step 4: Re-encode tail (last keyframe to OUT point)
ffmpeg -ss {keyframe_after_out} -to {out_point} -i input.mp4 -c:v libx264 tail.mp4

# Step 5: Concatenate
ffmpeg -f concat -safe 0 -i list.txt -c copy output.mp4
```

### Simplified approach (v0.8.0)
For the first version, use the simpler "snap to keyframe" approach:
1. Offer two export modes: "Fast (snap to keyframes)" and "Precise (re-encode)"
2. Fast mode: `-ss` before `-i` (input seeking to keyframe) with `-c copy`
3. Show the user how much the IN/OUT points shifted: "Adjusted to nearest keyframe: 2.3s → 2.0s"
4. Precise mode: current behavior (full re-encode)

### UI changes
- Add export mode toggle in export panel
- Show estimated export time for each mode
- Show keyframe positions on the timeline as subtle tick marks

**Verification:**
- Fast export a 30-second clip — should complete in under 2 seconds
- Compare fast vs precise export — fast should have slightly different start/end but identical quality in the middle
- Fast export should produce valid, playable MP4 files
- Audio tracks should be correctly aligned in fast mode

## Export Presets

**Files:** `src/ui/export.rs`, `src/config.rs`

**Problem:** Users export clips for different platforms with different requirements. Discord has a 25MB limit (or 50MB with Nitro). YouTube wants high quality. Twitter has resolution and duration limits. Currently, users manually adjust settings each time.

**Implementation:**

### Built-in presets
```rust
pub struct ExportPreset {
    pub name: String,
    pub format: String,        // mp4, webm, gif
    pub video_codec: String,   // copy, h264, h265, av1, vp9
    pub audio_codec: String,   // copy, aac, opus
    pub resolution: Option<(u32, u32)>,  // None = original
    pub max_bitrate_kbps: Option<u32>,
    pub max_file_size_mb: Option<u32>,
    pub fps: Option<u32>,      // None = original
    pub two_pass: bool,
}
```

### Presets
| Preset | Format | Codec | Resolution | Target |
|--------|--------|-------|------------|--------|
| **Original Quality** | MP4 | Copy | Original | Archive |
| **Discord (Free)** | MP4 | H.264 | 720p | 25 MB max |
| **Discord (Nitro)** | MP4 | H.264 | 1080p | 50 MB max |
| **YouTube** | MP4 | H.264 | Original | 20 Mbps |
| **Twitter/X** | MP4 | H.264 | 720p | 15 Mbps, 2:20 max |
| **GIF** | GIF | - | 480p | 15fps, 15s max |
| **Custom** | User-defined | | | |

### File size targeting
For Discord presets with a max file size:
1. Calculate target bitrate: `target_bitrate = (max_size_mb * 8 * 1024) / duration_seconds - audio_bitrate`
2. Use two-pass encoding for accurate file size
3. If the clip is too long to fit at minimum quality, show a warning: "Clip is too long for 25MB at acceptable quality. Consider trimming to under 60 seconds."

### UI
- Preset dropdown in the export panel (above the existing settings)
- Selecting a preset fills in all settings, grayed out to show they're preset-controlled
- "Customize" button unlocks individual settings
- Estimated file size shown live: "Estimated: 18.5 MB / 25 MB limit"
- After export, show actual file size and a comparison: "23.1 MB — fits Discord free tier"

**Verification:**
- Export with Discord Free preset — file should be under 25 MB
- Export a 5-minute clip with Discord Free — should show "too long" warning
- YouTube preset — should produce high quality output
- GIF preset — should produce a playable GIF under reasonable size

## Batch Export

**Files:** `src/ui/clips.rs`, `src/ui/export.rs`

**Problem:** Users who record multiple clips per session want to export them all at once with the same settings, not one at a time.

**Implementation:**

### Selection
1. Ctrl+click multiple clips in the clips library (already supported via `selected_clips`)
2. Right-click → "Export Selected" or use the existing "+" menu
3. Show a batch export panel with:
   - List of selected clips with thumbnails and durations
   - Shared export settings (preset, format, quality)
   - Per-clip trim (optional — default to full clip)
   - Output directory selector
   - Total estimated size and time

### Execution
1. Export clips sequentially (not in parallel — would overwhelm GPU encoder)
2. Show overall progress: "Exporting clip 3 of 7 — 42% overall"
3. Individual clip progress bar within the overall progress
4. Skip failed exports and continue with remaining clips
5. Summary at the end: "6 of 7 clips exported successfully. 1 failed: [error]"

### Naming
Output filenames: `{game_title}_{date}_{time}.mp4` or user-configurable template:
- `{game}` — game title
- `{date}` — YYYY-MM-DD
- `{time}` — HH-MM-SS
- `{duration}` — clip duration
- `{n}` — sequential number

**Verification:**
- Select 5 clips, batch export — all 5 should be exported
- Cancel mid-batch — completed exports should remain, pending ones cancelled
- One clip fails (e.g., source deleted) — rest should still export
- Progress should accurately reflect overall completion

## Direct Upload to Platforms

**Files:** New `src/upload.rs`, `src/ui/export.rs`

**Problem:** After exporting, users manually open Discord/YouTube/etc. and upload the file. This is friction that can be eliminated.

**Implementation:**

### Discord (webhook)
1. User provides a Discord webhook URL in settings
2. After export (or as an export option): upload the file via the webhook
3. Discord webhooks accept files up to 25 MB (free) or 50 MB (Nitro boosted servers)
4. Show upload progress and confirmation

```rust
async fn upload_to_discord(webhook_url: &str, file_path: &Path, message: &str) -> Result<()> {
    let client = reqwest::Client::new();
    let file = tokio::fs::read(file_path).await?;
    let form = reqwest::multipart::Form::new()
        .text("content", message.to_string())
        .part("file", reqwest::multipart::Part::bytes(file)
            .file_name(file_path.file_name().unwrap().to_string_lossy().to_string()));
    client.post(webhook_url).multipart(form).send().await?.error_for_status()?;
    Ok(())
}
```

### YouTube (OAuth)
1. Implement YouTube Data API v3 upload
2. OAuth2 flow: open browser for consent, receive token via localhost redirect
3. Upload with metadata: title, description, privacy (unlisted by default), tags
4. Show upload progress (YouTube API supports resumable uploads)
5. After upload, show the video URL

### File system "upload"
1. Copy the exported file to a user-specified directory (e.g., a synced cloud folder)
2. Useful for Dropbox, Google Drive, OneDrive users

### Initial scope (v0.8.0)
- Discord webhook upload (simplest, most requested)
- Copy to folder
- Defer YouTube API integration to v0.9.0 (requires OAuth complexity)

**Verification:**
- Discord: export and upload — file appears in the Discord channel within seconds
- Discord: file over 25 MB — should show an error with a suggestion to use a smaller preset
- Copy to folder: file appears in the target directory
- Upload progress should be accurate
- Network failure mid-upload — should show error, not crash

## Clip Thumbnails and Previews

**Files:** `src/utils.rs`, `src/ui/clips.rs`

**Problem:** Thumbnail generation currently picks a single frame (first frame or a frame at 1 second). This often captures a loading screen or black frame, making clips hard to identify in the library.

**Implementation:**

### Smarter thumbnail selection
1. Extract 5 candidate frames at 10%, 25%, 50%, 75%, 90% of the clip duration
2. Score each frame for "interestingness":
   - Reject mostly-black frames (average brightness < 10)
   - Reject mostly-uniform frames (low variance = loading screen)
   - Prefer frames with high edge content (more detail = gameplay)
3. Use the highest-scoring frame as the thumbnail

Simple scoring with ffmpeg:
```bash
ffmpeg -i clip.mp4 -vf "select=eq(n\,FRAME),metadata=print:file=-" -frames:v 1 -f null -
```

Or extract frames and score in Rust using the `image` crate:
```rust
fn score_frame(img: &DynamicImage) -> f64 {
    let gray = img.to_luma8();
    let mean: f64 = gray.pixels().map(|p| p[0] as f64).sum::<f64>() / gray.pixels().count() as f64;
    let variance: f64 = gray.pixels().map(|p| (p[0] as f64 - mean).powi(2)).sum::<f64>() / gray.pixels().count() as f64;
    if mean < 10.0 { return 0.0; } // black frame
    variance // higher variance = more interesting
}
```

### Animated thumbnails on hover
When the user hovers over a clip card, show a short animated preview:
1. Pre-generate a 3-second preview strip (10 frames at low resolution)
2. On hover, cycle through the frames at 3fps
3. Generate lazily — only when the clip card enters the viewport
4. Cache in a `thumbnails/` directory alongside the clip

**Verification:**
- Clips that start with black screens should show a gameplay frame as thumbnail
- Hover over a clip card — should see a short animated preview
- Thumbnail generation should not block the UI
- Clips with no good frames (all black) should show a generic placeholder
