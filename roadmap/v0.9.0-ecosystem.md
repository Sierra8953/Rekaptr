# Rekaptr v0.9.0 — Ecosystem

**Theme:** Transform Rekaptr from a standalone app into a complete product. Auto-updates, proper installation, crash reporting, and the infrastructure needed to support a growing user base.

**Target:** 5-6 weeks after v0.8.0

## Auto-Updater

**Files:** New `src/updater.rs`, `src/main.rs`, `src/ui/settings/general.rs`

**Problem:** Users running v0.1.0 have no way to receive fixes from v0.2.0+ without manually downloading and replacing files. For a recording app that handles important footage, running outdated versions with known bugs is a real risk.

**Implementation:**

### Update check
1. On app launch (and optionally every 24 hours), query a GitHub Releases API endpoint:
   ```
   GET https://api.github.com/repos/Sierra8953/Rekaptr-v2/releases/latest
   ```
2. Compare `tag_name` from the response against the compiled-in version (`env!("CARGO_PKG_VERSION")`)
3. If a newer version exists, show a non-blocking notification in the UI:
   ```
   Rekaptr v0.5.0 is available (you have v0.4.0)
   [Update Now]  [Release Notes]  [Skip This Version]  [Remind Me Later]
   ```

### Download and install
1. Download the release asset (`.zip` or `.msi`) to a temp directory
2. Show download progress in the UI
3. Verify the download with a SHA256 checksum (published in the release body or as a separate `.sha256` asset)
4. For `.zip` distribution:
   - Extract to a staging directory next to the app
   - Write a small `updater.bat` script that:
     1. Waits for the main process to exit
     2. Copies new files over old ones
     3. Relaunches the app
   - Launch `updater.bat` and exit the current process
5. For `.msi` distribution:
   - Launch the MSI installer silently (`msiexec /i update.msi /passive`)
   - The installer handles file replacement and relaunch

### Settings
```rust
pub enum UpdatePolicy {
    AutoCheck,          // Check and notify (default)
    AutoInstall,        // Download and install without asking
    ManualOnly,         // Never check automatically
}
```

Add to settings panel:
- Update policy dropdown
- "Check for Updates Now" button
- "Current version: v0.4.0"
- "Skip version v0.5.0" option (don't nag about a specific version)

### Security considerations
- Always verify checksums before applying updates
- Download only from the official GitHub releases URL (no redirect following to arbitrary domains)
- The update check respects the `check_for_updates` config flag (already exists in `AppConfig`)
- Never auto-install without user consent on the first update

**Verification:**
- Publish a test release with a higher version number — app should detect and offer update
- Download completes — checksum should match
- Apply update — new version launches correctly, settings preserved
- Skip version — that version is not offered again
- Disable auto-check — no network requests on launch

## Installer (WiX Toolset)

**Files:** New `installer/` directory, `build.rs` or CI scripts

**Problem:** Users currently receive a zip file and must manually place `ffmpeg.exe`, `ffprobe.exe`, and `libmpv-2.dll` in the `bin/` directory. Missing dependencies cause silent failures — no video playback, no thumbnail generation, no export. The setup wizard detects this but can't fix it.

**Implementation:**

### WiX v4 installer
Create a proper MSI installer that:
1. Installs Rekaptr to `C:\Program Files\Rekaptr\`
2. Bundles all dependencies:
   - `ffmpeg.exe` and `ffprobe.exe` (from gyan.dev or BtbN builds — both have LGPL-compatible licenses)
   - `libmpv-2.dll` (from mpv.io nightly builds)
   - GStreamer runtime (minimal subset: core, base plugins, d3d11 plugins, NVIDIA encoder plugins)
3. Creates Start Menu shortcut
4. Creates Desktop shortcut (optional, user-selectable)
5. Registers uninstaller in Add/Remove Programs
6. Registers `.luma` file association (for future project files)
7. Optionally adds "Open in Rekaptr" to right-click context menu for video files

### Installer UI
WiX supports custom UI with:
- Welcome page
- License agreement (display Rekaptr's license)
- Install location selector (default: `C:\Program Files\Rekaptr\`)
- Component selection:
  - [x] Rekaptr (required)
  - [x] FFmpeg tools
  - [x] Video playback (libmpv)
  - [ ] Desktop shortcut
  - [ ] Start with Windows
- Installation progress
- Finish page with "Launch Rekaptr" checkbox

### CI integration
Add a GitHub Actions workflow:
```yaml
build-installer:
  steps:
    - cargo build --release
    - Download ffmpeg, ffprobe, libmpv (from pinned URLs with checksum verification)
    - Run wix build installer/luma.wxs
    - Upload luma-setup-v0.9.0.msi as release artifact
```

### Portable mode
For users who don't want to install:
- Continue shipping the `.zip` distribution
- If a `portable.txt` file exists next to `luma.exe`, use exe-relative paths for everything (config, cache, recordings)
- The installer does NOT create `portable.txt`

### Dependency versioning
Pin exact versions of all bundled dependencies:
```
ffmpeg: 7.1 (gyan.dev release build)
ffprobe: 7.1 (same package)
libmpv: 0.38.0 (shinchiro build)
GStreamer: 1.24.x (minimal custom build or official MSVC runtime)
```

Store versions in a `dependencies.json` manifest for reproducibility.

**Verification:**
- Run installer on a clean Windows 11 VM — app should work immediately
- Uninstall via Add/Remove Programs — all files removed, settings in %APPDATA% preserved
- Install over a previous version — settings and recordings preserved
- Portable zip on a USB drive — works without installation

## Crash Reporting

**Files:** New `src/crash_reporter.rs`, `src/main.rs`

**Problem:** When the app crashes (panic, access violation, GPU driver crash), the user sees a Windows error dialog and loses their recording. There's no way for the developer to learn about crashes without users manually reporting them.

**Implementation:**

### Panic handler
```rust
fn setup_crash_reporter() {
    std::panic::set_hook(Box::new(|panic_info| {
        let payload = if let Some(s) = panic_info.payload().downcast_ref::<&str>() {
            s.to_string()
        } else if let Some(s) = panic_info.payload().downcast_ref::<String>() {
            s.clone()
        } else {
            "Unknown panic".to_string()
        };

        let location = panic_info.location().map(|l| format!("{}:{}:{}", l.file(), l.line(), l.column())).unwrap_or_default();

        let report = CrashReport {
            version: env!("CARGO_PKG_VERSION").to_string(),
            os_version: get_windows_version(),
            panic_message: payload,
            location,
            backtrace: std::backtrace::Backtrace::force_capture().to_string(),
            recording_active: was_recording_active(),
            gpu_info: get_gpu_info(),
            timestamp: chrono::Utc::now(),
        };

        // Save to disk (in case upload fails)
        save_crash_report_to_disk(&report);

        // Show a friendly dialog
        show_crash_dialog(&report);
    }));
}
```

### Crash dialog
Instead of the generic Windows crash dialog, show a custom window:
```
┌──────────────────────────────────────┐
│  Rekaptr has encountered an error       │
│                                      │
│  We're sorry — Rekaptr crashed          │
│  unexpectedly.                       │
│                                      │
│  Your recording has been saved up    │
│  to the last completed segment.      │
│                                      │
│  Would you like to send a crash      │
│  report to help us fix this?         │
│                                      │
│  [Send Report]  [Don't Send]         │
│                                      │
│  [ ] Always send crash reports       │
└──────────────────────────────────────┘
```

### Report submission
1. User opts in per-crash or via "always send" setting
2. Report sent as a POST to a crash reporting endpoint:
   - Option A: GitHub Issues API (create an issue with the crash label)
   - Option B: Self-hosted endpoint (e.g., Sentry, or a simple Cloudflare Worker that writes to a database)
   - Option C: Email via SMTP (simplest, no backend needed)
3. Report includes: version, OS, GPU, panic message, backtrace, whether recording was active
4. Report does NOT include: file paths, usernames, recording content, config details

### Recording recovery
When a crash occurs during recording:
1. The GStreamer pipeline is forcefully terminated (segments up to the last completed fragment are intact)
2. On next launch, detect incomplete sessions (segments without proper EOS fixup)
3. Offer to recover: "Rekaptr detected an incomplete recording from your last session. Would you like to recover it?"
4. Recovery: run `fixup_eos_segments` on the incomplete session

### Consent and privacy
- Crash reporting is opt-in (default: ask each time)
- "Always send" is a user choice, not a default
- Reports are stripped of PII before sending
- Settings panel shows: "Crash Reporting: [Ask Each Time / Always Send / Never Send]"
- Link to privacy policy explaining what data is collected

**Verification:**
- Force a panic with a test button — crash dialog should appear
- Send a test report — verify it arrives at the endpoint
- Crash during recording — next launch should offer recovery
- "Never Send" setting — no report dialog shown, crash just exits
- Crash report file saved locally even if upload fails

## Plugin / Extension API

**Files:** New `src/plugin.rs`, `src/plugin_api.rs`

**Problem:** Users have diverse needs that can't all be built into the core app: custom overlays, game-specific features, integration with streaming platforms, clip highlight detection, etc.

**Implementation:**

### Initial scope (v0.9.0) — Event hooks
Start with a simple event-based system, not a full plugin runtime:

1. **Event hooks:** Shell commands or scripts triggered by recording events
   ```toml
   # luma-hooks.toml
   [hooks]
   on_recording_start = "powershell -File notify_discord.ps1 'Recording started'"
   on_recording_stop = "powershell -File notify_discord.ps1 'Recording stopped'"
   on_clip_saved = "powershell -File upload_clip.ps1 '{clip_path}'"
   on_replay_saved = "powershell -File upload_replay.ps1 '{replay_path}'"
   ```

2. **Variables available to hooks:**
   - `{clip_path}` — full path to the saved clip
   - `{game_title}` — name of the recorded game
   - `{duration}` — recording duration in seconds
   - `{timestamp}` — Unix timestamp of the event
   - `{resolution}` — recording resolution

3. **Execution:** Hooks run asynchronously in the background, stdout/stderr logged

### Future scope (v1.0.0+) — Lua/WASM plugins
- Lua scripting for custom overlay rendering
- WASM modules for audio processing plugins
- Plugin marketplace (curated list on GitHub)

### Configuration UI
Add a "Hooks" section in settings:
- List of event types with associated commands
- Test button for each hook (fires the event with sample data)
- Log viewer showing recent hook executions and their output

**Verification:**
- Configure `on_clip_saved` hook — should fire after every clip export
- Hook script fails — should log error, not crash the app
- Variables should be correctly substituted
- Hooks should not block the UI or recording pipeline

## Documentation and Onboarding

**Files:** New `docs/` directory, in-app help system

**Problem:** New users have no guidance beyond the setup wizard. Features are discoverable only by clicking around. Common questions ("How do I record just one game?", "Why is my recording choppy?") have no in-app answers.

**Implementation:**

### In-app help
1. Add a `?` button in the top-right corner of each settings panel
2. Clicking opens a contextual help sidebar with:
   - Explanation of each setting
   - Recommended values for common scenarios
   - "Learn more" links to full documentation

### First-use tooltips
On first launch after install (or after a major version upgrade):
1. Show a brief guided tour highlighting key UI elements:
   - "Select a source to record" (points to source dropdown)
   - "Press F9 to start recording" (points to recording button)
   - "Your clips appear here" (points to clips tab)
2. User can dismiss or skip the tour
3. Tour only shows once (flag in config: `onboarding_completed_version`)

### Troubleshooting guide
Built into the app under Help menu:
- "Recording is choppy" → Check encoder settings, try H.264 instead of AV1, lower resolution
- "No audio in recording" → Check audio track configuration, verify device selection
- "Video playback is black" → Check HLS server status, verify ffmpeg is installed
- "Export fails" → Check ffmpeg installation, verify disk space
- "App won't start" → Delete config (backup first), reinstall GStreamer runtime

### Online documentation
GitHub Wiki or dedicated docs site:
- Getting started guide with screenshots
- Configuration reference (every setting explained)
- Troubleshooting FAQ
- Keyboard shortcuts reference
- API documentation for hooks

**Verification:**
- First launch shows guided tour
- Help button on each settings panel opens relevant content
- Troubleshooting guide covers the top 10 user issues
- Documentation is accurate for the current version
