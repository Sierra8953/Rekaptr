## Comprehensive Test Suite

**Files:** `tests/`, `src/**/tests.rs`

**Problem:** The codebase has minimal tests (a few unit tests for `resolve_pid` and config serialization). There are no integration tests, no UI tests, and no recording pipeline tests. Regressions are caught by manual testing only.

**Implementation:**

### Unit tests (expand existing)
Target the core logic modules with pure-function tests:

**`src/utils.rs`**
- `clean_title()` — edge cases: unicode, empty string, very long names, path separators, collisions
- `compute_total_duration()` — empty directory, single segment, many segments, segments with no duration suffix
- `fixup_eos_segments()` — no segments, already-fixed segments, corrupt filenames, locked files
- `scan_segments()` — empty dir, duplicate indices, mixed session IDs, sort order
- `generate_master_playlist()` — single segment, multi-session with discontinuities, very long recordings

**`src/config.rs`**
- `AppConfig` serialization round-trip (already exists, expand)
- Migration runner: version 0 → 1, version 1 → 1 (no-op), fresh database
- `load_favorites()` / `set_favorite()` — add, remove, duplicate, empty DB
- `get_db_path()` — with and without APPDATA, portable mode

**`src/engine.rs`**
- `generate_pipeline_string()` — all encoder types, all rate control modes, with/without HWND, with/without audio tracks, edge case: zero audio tracks
- `resolve_pid()` — empty input, "Default", nonexistent process, process with visible window
- `validate_encoder()` — known encoders, bogus encoder name

**`src/db.rs`**
- Full CRUD cycle: register session, update duration, query, verify
- Concurrent access (two GameDatabase instances on same file)
- Migration from version 0

### Integration tests
Test multi-component flows without a GUI:

**Recording lifecycle**
```rust
#[test]
fn test_recording_lifecycle() {
    gst::init().unwrap();
    let config = AppConfig::default();
    let pipeline_str = PipelineBuilder::new(&config.global_video, ...)?.build();
    let pipeline = gst::parse::launch(&pipeline_str)?;

    // Start
    assert_eq!(pipeline.set_state(gst::State::Playing), Ok(StateChangeSuccess::Async));

    // Wait for first fragment
    let bus = pipeline.bus().unwrap();
    let msg = bus.timed_pop_filtered(gst::ClockTime::from_seconds(10), &[gst::MessageType::Element]);
    assert!(msg.is_some()); // fragment-opened or fragment-closed

    // Stop
    pipeline.send_event(gst::event::Eos::new());
    let eos = bus.timed_pop_filtered(gst::ClockTime::from_seconds(5), &[gst::MessageType::Eos]);
    assert!(eos.is_some());
    pipeline.set_state(gst::State::Null)?;

    // Verify segments exist
    assert!(game_dir.join("seg_0000000000_*.m4s").exists());
}
```

### Stress tests
Run as part of CI but with a longer timeout:

- **Long recording:** Record for 1 hour continuously, verify all segments are intact
- **Rapid start/stop:** Start and stop recording 100 times in 60 seconds, verify no file corruption or leaked threads
- **Low disk:** Fill disk to 99%, start recording, verify graceful handling (toast, auto-stop)
- **Low memory:** Limit process to 512MB, record at 4K — verify OOM handling
- **GPU reset:** Simulate GPU driver crash during recording (TDR), verify recovery or clean error

## Accessibility Audit

**Files:** All UI files (`src/ui/*.rs`)

**Problem:** The app has no accessibility considerations. Screen readers can't navigate it, keyboard-only users can't operate it, and color-blind users may miss color-coded indicators.

**Implementation:**

### Keyboard navigation
1. All interactive elements must be reachable via Tab key
2. Tab order should follow visual layout (left-to-right, top-to-bottom)
3. Focused elements should have a visible focus ring
4. Enter/Space activates the focused element
5. Escape closes modals and popovers
6. Arrow keys navigate within groups (sidebar items, clip grid, settings tabs)

### Screen reader support
1. Add ARIA-equivalent labels to all interactive elements (GPUI's accessibility API)
2. Announce state changes: "Recording started", "Recording stopped", "Clip saved"
3. Describe non-text content: "Clip thumbnail: Counter-Strike 2, 2 minutes 30 seconds, January 15 2026"
4. Live regions for dynamic content: recording timer, audio levels, export progress

### Color and contrast
1. All text must meet WCAG 2.1 AA contrast ratio (4.5:1 for normal text, 3:1 for large text)
2. Don't rely solely on color to convey information:
   - Recording indicator: red dot + "REC" text + pulsing animation
   - Audio levels: color bars + dB numbers
   - Clip favorited: star icon + "Favorited" label (not just yellow color)
3. Test with Windows High Contrast mode — UI should remain usable

### Motion and animation
1. Respect Windows "Reduce motion" setting (`SPI_GETCLIENTAREAANIMATION`)
2. If enabled: disable pulsing recording indicator, tooltip fade-ins, hover animations
3. Keep functional animations (progress bars, loading spinners) but reduce decorative ones

**Verification:**
- Navigate the entire app using only keyboard — every feature should be reachable
- Enable Windows Narrator, perform common tasks — all actions should be announced
- Enable High Contrast mode — all UI elements visible and functional
- Run an automated contrast checker on all color combinations

## Localization Framework

**Files:** New `locales/` directory, all UI files

**Problem:** All UI strings are hardcoded in English. Supporting other languages requires a localization framework, even if v1.0.0 only ships with English.

**Implementation:**

### String extraction
1. Replace all hardcoded UI strings with localization keys:
   ```rust
   // BEFORE:
   .child("Clips Library")

   // AFTER:
   .child(t!("clips.library_title"))
   ```

2. Create a `t!()` macro that looks up the key in the current locale's string table

### Locale files
Use a simple format (TOML or Fluent):
```toml
# locales/en-US.toml
[clips]
library_title = "Clips Library"
search_placeholder = "Search clips..."
no_clips_title = "No clips found"
no_clips_subtitle = "Start recording to see your clips here"

[recording]
start = "Start Recording"
stop = "Stop Recording"
paused = "Paused"

[export]
title = "Export Clip"
progress = "Exporting... {percent}%"
success = "Export complete"
```

### Runtime locale selection
1. Default: follow Windows system locale (`GetUserDefaultLocaleName`)
2. Override in settings: language dropdown
3. Hot-reload: changing language updates all strings without restart (re-render)

### Initial scope (v1.0.0)
- Framework in place with `t!()` macro
- English (en-US) locale file with all strings extracted
- Settings panel for language selection (only English available initially)
- Contribution guide for translators (how to add a new locale file)

**Verification:**
- All UI text comes from the locale file (no hardcoded strings remaining)
- Changing locale in settings updates the UI immediately
- Missing translation keys fall back to English
- Strings with variables (e.g., "{percent}%") render correctly

## Final Performance Pass

**Files:** All performance-critical paths

**Problem:** By v1.0.0, features have accumulated across 8 releases. Each release optimized its own additions, but the cumulative impact needs a holistic review.

**Implementation:**

### Profiling session
1. Record a profiling session covering the full user flow:
   - App launch → source selection → start recording → record 5 minutes → stop → browse clips → preview clip → export clip
2. Capture with:
   - `tracy` profiler for frame timing and CPU hotspots
   - Windows Performance Recorder for GPU and I/O
   - Process Explorer for thread count and memory

### Target metrics
| Metric | Target | How to measure |
|--------|--------|---------------|
| App launch to interactive | < 2 seconds | Stopwatch from exe launch to first paint |
| Recording start latency | < 500ms | From hotkey press to first frame captured |
| Frame drop rate | < 0.1% at 60fps | GStreamer QoS stats |
| Memory usage (idle) | < 150 MB | Task Manager |
| Memory usage (recording) | < 300 MB | Task Manager |
| Memory growth over 1 hour | < 50 MB | Monitor every 5 minutes |
| Timeline scrub responsiveness | < 16ms per frame | tracy frame timing |
| Export speed (copy mode) | > 100x realtime | Time a 60s clip export |
| Clip library load (1000 clips) | < 1 second | Stopwatch |

### Common optimization targets
- Unnecessary allocations in render loops (clone, to_string, format!)
- Texture uploads that could be zero-copy
- SQLite queries that could be batched
- Thread spawning that could use a pool
- String building that could use a pre-allocated buffer

### Memory leak detection
1. Run the app for 8 hours with continuous recording start/stop cycles
2. Monitor RSS every minute
3. Any linear growth indicates a leak
4. Use DHAT or Windows Debug Diagnostic Tool to identify leak sources

**Verification:**
- All target metrics met on a mid-range system (RTX 3060, 16GB RAM, NVMe SSD)
- All target metrics met on a low-end system (GTX 1650, 8GB RAM, SATA SSD)
- No memory leaks after 8-hour stress test

## Distribution Pipeline

**Files:** `.github/workflows/`, installer configuration

**Problem:** Releasing a new version requires manual steps: build, package, upload, announce. This is error-prone and slow.

**Implementation:**

### Automated release workflow
```yaml
name: Release
on:
  push:
    tags: ['v*']

jobs:
  build:
    runs-on: windows-latest
    steps:
      - checkout
      - Install GStreamer MSVC runtime
      - Install WiX Toolset
      - cargo build --release
      - Download and verify ffmpeg/ffprobe/libmpv
      - Build MSI installer
      - Build portable ZIP
      - Generate SHA256 checksums

  release:
    needs: build
    steps:
      - Create GitHub Release from tag
      - Upload MSI installer
      - Upload portable ZIP
      - Upload checksums
      - Generate changelog from git log since last tag

  notify:
    needs: release
    steps:
      - Post to Discord announcement channel (webhook)
      - Update version check endpoint
```

### Release checklist (manual)
Before tagging a release:
1. [ ] All tests pass on CI
2. [ ] Version bumped in `Cargo.toml`
3. [ ] Changelog updated in `CHANGELOG.md`
4. [ ] Tested on clean Windows 11 VM (installer + fresh launch)
5. [ ] Tested upgrade from previous version (settings preserved)
6. [ ] Tested on Windows 10 (minimum supported version)
7. [ ] No compiler warnings (`cargo clippy -- -D warnings`)
8. [ ] License headers present on all source files
9. [ ] Dependencies audited (`cargo audit`)
10. [ ] Binary signed with code signing certificate (if available)

### Versioning
Follow semantic versioning strictly:
- **MAJOR** (1.0.0): Breaking changes to user data formats, config schema, or plugin API
- **MINOR** (0.X.0): New features, non-breaking changes
- **PATCH** (0.0.X): Bug fixes only

**Verification:**
- Push a tag `v1.0.0-rc1` — CI builds and uploads release artifacts
- Download the MSI from the release — installs and runs correctly
- Auto-updater detects the RC release (if update channel is set to "beta")
- Changelog is accurate and complete
