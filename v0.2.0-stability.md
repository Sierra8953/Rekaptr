# Rekaptr v0.2.0 — Stability

**Theme:** Fix every bug that causes crashes, data loss, or silent failures. After this release, Rekaptr should never crash during normal operation.

**Target:** 2-3 weeks after v0.1.0

### Changes

- **#1** `src/video_player/video.rs` — transmuted `ID3D11Device` is now wrapped in `ManuallyDrop` so the COM wrapper's `Release()` no longer fires when the function-scope binding drops. AppState's single AddRef remains the sole reference; previously every Video creation silently decremented it.
- **#2** `src/ui/recording.rs` — `set_state(Playing)` now checks the result, drains the bus for a real error, tears down the pipeline + mic subscribers, returns phase to `Idle`, and surfaces an error toast.
- **#3** `src/state.rs`, `src/ui/recording.rs` — `RecordingState` gained a `teardown_in_progress: AtomicBool`. The background teardown thread sets it true before running and clears it after `fixup_eos_segments` completes. `toggle_recording_ext` refuses to start a new recording (with a "please wait" toast) while the flag is set, closing the window where phase could flip to `Idle` before files were finalized.
- **#4** `src/ui/recording.rs` — emergency `toggle_recording_internal` now sends EOS first and waits up to 500ms on the bus for `Eos`/`Error` before forcing `set_state(Null)`, giving `splitmuxsink` a chance to finalize the last fragment's `mfra`/`mdat`.
- **#5** `src/main.rs`, `src/ui/mod.rs` — HLS server probes ports 8080–8089, falls back to an ephemeral port if all are taken, stores the chosen port in a global `AtomicU16`, and URL/CORS builders read it via `get_hls_port()`.
- **#6** `src/state.rs`, `src/ui/recording.rs` — `RecordingState` gained `mic_subscriber_keys`; a new `clear_mic_subscribers()` runs on stop, on emergency teardown, on pipeline-launch failure, and on `set_state(Playing)` failure.
- **#7** `src/config.rs` — added a module-level `OnceLock<RwLock<Option<AppConfig>>>`. `AppConfig::load()` returns a clone of the cached value (in-memory) instead of opening SQLite on every render frame; `AppConfig::save()` writes through to both disk and cache. Disk I/O split into a private `load_from_disk` / `write_to_disk` pair so the cache stays the only entry point.
- **#8** `src/video_player/video.rs` — `static mut OPENGL32_DLL` replaced with `OnceLock<OpenGl32Module>` (Send/Sync newtype around `HMODULE`); reads go through `opengl32_handle()`.

## Critical: COM Pointer Double-Free in Video Player

**File:** `src/video_player/video.rs:127`

**Problem:** `std::mem::transmute_copy` converts a raw `*mut c_void` into an `ID3D11Device` COM smart pointer. The COM wrapper calls `Release()` on drop. If `Video::new_with_options` fails at any point after line 127 (e.g., `CreateTexture2D` fails, OpenGL context creation fails), the error path drops the `ID3D11Device`, which calls `Release()` on what was supposed to be a borrowed pointer from the GPUI renderer. This is a use-after-free that can crash the app or corrupt the D3D11 device state for the entire process.

**How it manifests:** Intermittent crash when opening clip preview, especially when GPU is under load or when switching between clips rapidly. The crash dump will show an access violation inside `d3d11.dll`.

**Fix:**
1. Wrap the transmuted device in `ManuallyDrop<ID3D11Device>` to prevent automatic Release on drop
2. Only extract the inner value (via `ManuallyDrop::into_inner`) after all fallible operations have succeeded and the device is safely stored in the `VideoInner` struct
3. Add an explicit `AddRef` call if the device needs to outlive the function scope independently

**Verification:**
- Open and close the clip preview 50+ times rapidly
- Open preview while a recording is active (GPU contention)
- Force an error by temporarily breaking the OpenGL context setup — app should show error toast, not crash

## Critical: Pipeline State Failure Silently Ignored

**File:** `src/ui/recording.rs:191`

**Problem:** `let _ = pipeline.set_state(gst::State::Playing)` discards the return value. If the pipeline fails to enter Playing state (missing encoder, hardware busy, permission denied), the code continues to line 198 where `RecordingPhase` is set to `Recording`. The user sees the recording UI (timer, stats) but no data is being captured. They discover the problem only when they try to play back an empty recording.

**Possible causes of failure:**
- NVIDIA encoder in use by another application (e.g., OBS, ShadowPlay)
- Screen capture API denied by DRM-protected content
- Audio device disconnected between pipeline creation and state change
- GStreamer element negotiation failure (incompatible caps)

**Fix:**
1. Check the `StateChangeReturn` value from `set_state(Playing)`
2. If it returns `Failure`, extract the error from the pipeline bus
3. Show a descriptive error toast (e.g., "Recording failed: NVIDIA encoder is in use by another application")
4. Set phase back to `Idle` and clean up the pipeline
5. Also check `set_state` in the `Paused` transition if doing a two-step state change

**Code sketch:**
```rust
match pipeline.set_state(gst::State::Playing) {
    Ok(gst::StateChangeSuccess::Success | gst::StateChangeSuccess::Async) => {
        *self.app_state.recording.phase.lock() = RecordingPhase::Recording;
    }
    _ => {
        if let Some(bus) = pipeline.bus() {
            if let Some(msg) = bus.pop_filtered(&[gst::MessageType::Error]) {
                // Extract and display error
            }
        }
        let _ = pipeline.set_state(gst::State::Null);
        self.show_toast("Recording Failed", &error_msg, ToastLevel::Error, cx);
        return;
    }
}
```

**Verification:**
- Temporarily rename the encoder element in config to a bogus name — should get a clear error
- Start OBS with NVENC active, then try to record in Rekaptr — should get "encoder busy" error
- Disconnect audio device mid-startup — should get audio device error

## Critical: Recording Stop/Start Race Condition

**File:** `src/ui/recording.rs:41-76`

**Problem:** `stop_recording()` spawns a background thread for pipeline teardown (EOS, set_state(Null), fixup_eos_segments) but sets the phase to `Stopping` and then immediately notifies the UI. The phase transitions to `Idle` inside the background thread (line 60). If the user clicks "Record" again before the background thread finishes:
1. `toggle_recording_ext` reads phase as `Stopping` and returns early (line 36) — this is the safe case
2. But if timing is unlucky and the background thread sets `Idle` between the user's click and the phase check, a new recording starts while `fixup_eos_segments` is still running on the old session's files

**Additional race:** The auto-record feature (`src/main.rs:501`) polls the recording phase from a separate async task. It could read `Idle` and trigger `toggle_recording_ext` while teardown is in progress.

**Fix:**
1. Add a `Stopping` → `Idle` transition that goes through the UI thread via `cx.notify()` or a channel
2. Do not allow starting a new recording until the teardown thread has signaled completion
3. Use an `AtomicBool` or `oneshot` channel for the teardown thread to signal completion
4. In `start_recording`, check that no teardown is in progress

**Verification:**
- Record for 5 seconds, stop, immediately click record again — should not start until teardown finishes
- Write an automated test that calls stop/start in a tight loop 100 times — no panics or file corruption

## Critical: Emergency Stop Corrupts Last Segment

**File:** `src/ui/recording.rs:10-23`

**Problem:** `toggle_recording_internal()` calls `pipeline.set_state(gst::State::Null)` directly without sending EOS first. This is called from the GStreamer bus error handler (line 337) when the pipeline encounters a fatal error. The last segment being written by `splitmuxsink` is abruptly terminated without proper container finalization. The resulting `.m4s` file has:
- Missing `mfra` (movie fragment random access) box
- Potentially truncated `mdat` box
- Incorrect duration metadata

This segment will fail to play in mpv or may cause the entire HLS playlist to break because the segment duration doesn't match the manifest.

**Fix:**
1. Even in the error path, attempt to send EOS with a short timeout (500ms)
2. If EOS times out, then force `set_state(Null)`
3. After teardown, mark the last segment as potentially corrupt (rename with a `.partial` suffix or delete it)
4. `fixup_eos_segments` should detect and handle partial segments

**Verification:**
- Kill the NVIDIA driver during recording (use `devcon` to disable/enable the GPU adapter)
- Simulate a GStreamer error by unplugging the monitor during screen capture
- Verify the corrupt segment is handled gracefully and doesn't break playback of the rest

## Critical: Hardcoded HLS Port 8080 With No Fallback

**File:** `src/main.rs:561`

**Problem:** `TcpListener::bind("127.0.0.1:8080")` fails if port 8080 is already in use. This is extremely common — web developers run local servers on 8080, Docker often maps to it, and other media applications use it. When the bind fails:
1. The error is logged but the app continues normally
2. The HLS server never starts
3. All video playback silently fails — the timeline shows nothing, clips show black thumbnails
4. The user has no indication of what went wrong

**Fix:**
1. Try binding to a range of ports: 8080, 8081, ..., 8089
2. Store the successfully bound port in `AppState` (e.g., `hls_port: AtomicU16`)
3. Update all HLS URL construction (`src/ui/mod.rs:868`) to use the dynamic port
4. If all ports fail, show a prominent error in the UI: "Video playback unavailable — ports 8080-8089 are in use"
5. Consider using port 0 (OS-assigned ephemeral port) as the ultimate fallback

**Verification:**
- Start a simple HTTP server on port 8080 before launching Rekaptr — should auto-select 8081
- Block ports 8080-8089, launch Rekaptr — should show a clear error message
- Verify video playback works on the non-default port

## High: Mic Provider Subscriber Leak

**File:** `src/ui/recording.rs:246`

**Problem:** When recording starts, `setup_audio_feeders` inserts `AppSrc` references into `mic_provider.subscribers` DashMap. When recording stops, these are never removed. The mic provider thread (`src/audio.rs:95-130`) continues pushing audio buffers to these stale AppSrc sinks every ~20ms. Each push to a dead AppSrc returns an error that is silently ignored.

**Impact:**
- CPU waste: the mic provider processes audio for N dead subscribers after N recording sessions
- Memory: each AppSrc holds internal buffer queues that accumulate data
- After many start/stop cycles, measurable CPU and memory growth

**Fix:**
1. In `stop_recording()`, clear the mic provider subscribers: `self.app_state.mic_provider.subscribers.clear()`
2. Alternatively, store the subscriber keys used during setup and remove only those
3. In the mic provider push loop, remove subscribers that return errors on push

**Verification:**
- Start/stop recording 20 times, check memory usage — should be stable
- Monitor CPU usage of the mic provider thread after multiple sessions — should not increase

## High: Config Loaded From SQLite ~60 Times Per Second

**File:** `src/config.rs` (called from 60+ locations across 16 files)

**Problem:** `AppConfig::load()` opens a new SQLite connection, creates tables if not exist, queries the config row, and deserializes JSON — every single time it's called. Hot paths include:
- `get_current_audio_tracks()` (`src/ui/mod.rs:800`) — called every render frame during timeline display
- `get_storage_root()` (`src/utils.rs:50`) — called from the cleanup thread on every filesystem event
- Various settings panels re-read config on every render

At 60fps, this means ~60 SQLite connections opened and closed per second, plus JSON deserialization overhead. On HDDs or with antivirus scanning, this causes visible UI jank.

**Fix:**
1. Add a `Mutex<AppConfig>` (or `ArcSwap<AppConfig>`) to `AppState` as a cached config
2. Load once at startup, cache in memory
3. `AppConfig::load()` reads from the cache
4. `AppConfig::save()` writes to SQLite AND updates the cache
5. All mutation paths already go through `save()`, so the cache stays consistent
6. For the background cleanup thread, pass the storage path as a value instead of re-reading config

**Verification:**
- Profile before/after with `tracy` or `perf` — SQLite calls should drop to near-zero during normal operation
- Timeline scrubbing should feel noticeably smoother on systems with slow storage

## High: Static Mutable OPENGL32_DLL Without Synchronization

**File:** `src/video_player/video.rs:24`

**Problem:** `static mut OPENGL32_DLL: HMODULE` is read and written without any synchronization primitive. If two `Video::new_with_options` calls execute concurrently (e.g., loading a clip preview while the main timeline is initializing), both threads could race on the `LoadLibraryW` call. Additionally, `get_proc_address_callback` reads this static from the GStreamer/mpv render thread.

**Fix:**
Replace with `static OPENGL32_DLL: OnceLock<HMODULE>` which handles the initialization race safely:
```rust
static OPENGL32_DLL: std::sync::OnceLock<HMODULE> = std::sync::OnceLock::new();

fn get_opengl32() -> HMODULE {
    *OPENGL32_DLL.get_or_init(|| unsafe { LoadLibraryW(w!("opengl32.dll")).unwrap_or_default() })
}
```

**Verification:**
- Open two clip previews simultaneously (if the UI supports it)
- No UB under Miri/ThreadSanitizer (if applicable)
